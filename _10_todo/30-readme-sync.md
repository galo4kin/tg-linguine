# Шаг 30. Актуализация README по текущему состоянию кода

## Цель
Привести `README.md` к точному соответствию реально работающему боту
на mac mini: setup, переменные окружения, разворачивание, миграции
(если есть), список доступных команд бота. Документация — единственный
артефакт «релиза».

## Контекст
Релизного процесса как такового нет: бот живёт прямо в
`~/Projects/tg-linguine`, watchdog (шаг 14) перезапускает его каждую
минуту, новые версии раскатываются через `git pull && make build`.
README должен это отражать без вранья.

К моменту выполнения этой задачи в коде должны существовать как
минимум:
- env-переменные из `internal/config/config.go`;
- миграции в `migrations/` (применяются автоматически на старте через
  `embed` + `golang-migrate` в `cmd/bot/main.go`);
- slash-команды, зарегистрированные в `internal/telegram/bot.go` —
  `/start`, `/setkey`, и (при выполненных шагах 32–33) `/stats`,
  `/shutdown`, `/whoami`.

## Действия
1. Просканировать код:
   - `internal/config/config.go` — собрать список всех env-переменных
     с их default'ами и описаниями; сверить с `.env.example` 1:1.
   - `migrations/` — посмотреть, что есть, и как они применяются (по
     `cmd/bot/main.go`); описать в README одной фразой: «миграции
     эмбеддятся в бинарь и накатываются автоматически на старте».
   - `internal/telegram/bot.go` (и `internal/telegram/handlers/`) —
     собрать актуальный список slash-команд и не-slash-входов (URL,
     ввод ключа, callback-кнопок). Учесть только то, что реально
     зарегистрировано в коде на момент работы.
2. Перестроить разделы README в таком порядке:
   - короткое описание + стек;
   - Требования (Go-версия — взять реальную из `go.mod`; токен бота);
   - Быстрый старт (`make build`, `cp .env.example .env`, `make run`);
   - Переменные окружения — таблица, точно соответствующая
     `internal/config/config.go`;
   - Миграции — короткий абзац;
   - Доступные команды бота — таблица: команда → что делает → кому
     доступна (все / админ);
   - Деплой на mac mini — секция от шага 14, обновить, если что-то
     поменялось (например, упомянуть `/shutdown` как способ
     перезапуска вместо `kill -9`, если шаг 32 закрыт);
   - Master-ключ шифрования — оставить.
3. Удалить из README любые устаревшие фрагменты, которые не
   соответствуют коду (несуществующие env, команды, скрипты).
4. Сверить `.env.example` с `internal/config/config.go`: все env
   присутствуют, имена и default'ы совпадают, есть осмысленные
   комментарии.
5. Обновить `RELEASES.md`: добавить запись `## 30 — readme-sync` с
   кратким описанием, что синхронизировано.

## DoD
- Каждая env-переменная из `internal/config/config.go` упомянута и в
  README, и в `.env.example`.
- Каждая slash-команда, зарегистрированная в `internal/telegram/`,
  есть в таблице «Доступные команды бота» с пометкой `все`/`админ`.
- Раздел «Миграции» отражает реальный механизм запуска (auto on
  start).
- В README нет ссылок на несуществующие env, команды или скрипты
  (проверить grep'ом).
- `make build` + `make test` зелёные (Go-кода эта задача не меняет,
  но прогон обязателен по правилам проекта).
- Запись о шаге добавлена в `RELEASES.md`.

## Зависимости
Шаг 29 (последний кодовый шаг основной фазы). Имеет смысл прогнать
повторно после 32, чтобы захватить новые админ-команды; либо
дополнять README прямо в рамках шагов 31/32/33 и закрыть 30 после.
