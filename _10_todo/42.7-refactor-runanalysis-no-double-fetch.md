# Step 42.7 — refactor-runanalysis-no-double-fetch

## Откуда

Code review шага 40, finding §3: [_30_done/40-groq-respect-retry-after-review.md](_30_done/40-groq-respect-retry-after-review.md).

## Проблема

`AnalyzeArticle` (`internal/articles/usecase.go:198-207`) уже валидирует `users.ByID` и `keys.Get(ProviderGroq)`. Затем зовёт `runAnalysis` (`usecase.go:421-428`), который снова делает `s.users.ByID()` и `s.keys.Get()` — лишний round-trip к БД и повторная расшифровка ключа. То же самое в `AnalyzeExtracted` для `ModeSummarize` (l. 364-370).

## Что делаем

Привести к единой модели «верхний слой готовит, runAnalysis потребляет»:

1. Изменить сигнатуру `runAnalysis`:
   ```go
   func (s *Service) runAnalysis(
       ctx context.Context,
       userID int64,
       user *users.User,
       key string,
       languageCode, userCEFR string,
       extracted Extracted,
       notice string,
       onProgress ProgressFunc,
       start time.Time,
       tokensEstimated int,
   ) (*AnalyzedArticle, error)
   ```
2. `AnalyzeArticle`: сохранить уже полученные `user` и `key` (сейчас отбрасываются через `_, err := ...`), пробросить в `runAnalysis`. Убрать оттуда повторные fetch'и.
3. `AnalyzeExtracted`: для обоих режимов резолвить user+key один раз перед свитчем mode (а не только в `ModeSummarize`), пробросить в `runAnalysis`.
4. Маппинг `users.ErrNotFound → ErrNoActiveLanguage / ErrNoAPIKey` остаётся на верхнем слое — `runAnalysis` теперь fail-fast'ит, если кто-то прислал nil.
