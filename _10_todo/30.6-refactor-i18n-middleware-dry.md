# Рефакторинг 30.6. Дедуплицировать `langFromUpdate` в `bot.go`

## Контекст
Шаг 29 ввёл `langFromUpdate(*models.Update) string` в
`internal/telegram/bot.go:194-204` для panic-recovery middleware.
Чуть ниже (`bot.go:218-229`) `i18nMiddleware` повторяет тот же
switch in-line:

```go
lang := "en"
if update.Message != nil && update.Message.From != nil {
    lang = update.Message.From.LanguageCode
} else if update.CallbackQuery != nil {
    lang = update.CallbackQuery.From.LanguageCode
}
```

Две копии решают одну задачу — если завтра появится новый источник
(business-message, edited message и т.д.), баг проявится в одной из
веток, а не в обеих.

## Цель
Одна точка правды для извлечения языка пользователя из update.

## Действия
1. Заменить in-line switch в `i18nMiddleware` на вызов
   `langFromUpdate(update)`.
2. Проверить, что поведение совпадает: обе ветки возвращают `"en"`
   как fallback (хелпер делает это явно). Если есть тонкий нюанс —
   зафиксировать в `langFromUpdate`, не оставлять разнобоя.
3. Никаких новых публичных API не добавлять — `langFromUpdate`
   остаётся package-local.

## DoD
- В `bot.go` определение языка пользователя живёт ровно в одном
  месте (`langFromUpdate`).
- `i18nMiddleware` использует этот хелпер.
- `make build` и `make test` зелёные.

## Зависимости
Шаг 29 (источник `langFromUpdate`).
