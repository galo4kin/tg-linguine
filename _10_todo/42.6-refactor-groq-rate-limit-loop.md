# Step 42.6 — refactor-groq-rate-limit-loop

## Откуда

Code review шага 40, finding §2: [_30_done/40-groq-respect-retry-after-review.md](_30_done/40-groq-respect-retry-after-review.md).

## Проблема

`chat()` (`internal/llm/groq/analyze.go:111-151`) и `chatPlainText()` (`internal/llm/groq/summarize.go:53-89`) — почти полная копия retry-loop. Отличается только тип возврата (`[]byte` vs `string`), форма chatRequest (json_object vs без response_format) и лог-префикс. Сам цикл, clamp `wait > maxRetryAfter`, ctx.Done — продублированы. Clamp `retryAfterBuffer + maxRetryAfter` повторяется в обоих местах, хотя `parseRateLimitRetryAfter` уже клампит через `clampRetryAfter`.

## Что делаем

1. Вынести retry-loop в общий helper в `internal/llm/groq/client.go`:
   ```go
   func (c *Client) withRateLimitRetry(
       ctx context.Context,
       once func() (raw []byte, retryAfter time.Duration, err error),
       logPrefix string,
   ) ([]byte, error)
   ```
   Helper отвечает за лог `<logPrefix> rate-limit retry`, sleep с буфером и clamp, ctx.Done, возврат `ErrRateLimited` после исчерпания попыток.
2. `chat` и `chatPlainText` сворачиваются до вызова `c.withRateLimitRetry(ctx, c.chatOnce-or-chatPlainTextOnce, "groq.chat" / "groq.summarize")`.
3. Single-source clamp: вычисление `wait + retryAfterBuffer` и его clamp до `maxRetryAfter` живёт ровно в одном месте.
4. Тесты: проверить, что обновлённые функции по-прежнему уважают Retry-After (live-tests на mock'e не нужны — поведение покрыто `retry_after_test.go` для парсера; добавить unit-тест на сам loop с инжектируемым `once` если уйдёт быстро).
