# Refactor: общий «open stored article card»

## Цель
Свести три копии одного и того же сценария «достать статью → при
необходимости догенерировать адаптацию → перерисовать карточку» в один
helper в `internal/telegram/handlers/`.

## Контекст
После шагов 16/17/18/19 в трёх местах появился почти идентичный код:
- `internal/telegram/handlers/card_handler.go::HandleCallback`
- `internal/telegram/handlers/history.go::openArticle`
- `internal/telegram/handlers/url.go::Handle` (ветка после
  `AnalyzeArticle`, особенно после cache-hit'а — тоже надо проверить
  отсутствие текущего уровня и догенерировать)

Каждое место заново делает: `articles.ByID` → проверка владельца →
получение активного `CEFRLevel` → `resolveAbsoluteLevel` → если
`adapted_versions[abs]` пуст, edit-message «🪄 Адаптирую под X…»,
вызов `articles.Service.Adapt`, перезагрузка статьи → подсчёт
`CountByArticle` → `PageByArticle(preview)` → `renderArticleCard` +
`articleCardKeyboard` + `EditMessageText`.

Расхождения:
- url-handler в случае ошибки regen НЕ сообщает пользователю — пишет
  только в лог;
- card_handler/history — сообщают через i18n;
- preview в url-handler берётся из in-memory `result.Words`, в
  card_handler/history — из репо.

## Что сделать
1. Завести в `internal/telegram/handlers/` новый файл (например
   `card_render.go`) с хелпером вида
   ```go
   func renderStoredArticle(
       ctx context.Context, b *bot.Bot,
       deps cardRenderDeps,
       chatID any, msgID int,
       loc *goi18n.Localizer,
       userID int64, userCEFR string,
       article *articles.Article,
       view ArticleView,
       previewSource previewProvider, // either inline-words or repo lookup
   ) error
   ```
   с одной реализацией regen→reload→render.
2. Перевести `card_handler.go::HandleCallback`,
   `history.go::openArticle` и `url.go::Handle` на этот хелпер. В
   url-handler-е добавить ту же ветку «edit-message с локализованной
   ошибкой при regen failure», что и в остальных местах.
3. Подумать, не пора ли удалить `articles.Service.ArticleByID` —
   эта тонкая обёртка появилась в шаге 19 чтобы url-handler не
   тащил отдельный `articles.Repository`. После рефактора у хелпера
   будет одна точка входа в репо, и обёртка может уйти.
4. Тесты: обновить `card_test.go` если нужно; в идеале — табличный
   тест на хелпер с фейковым LLM/репозиторием (вариант: статья без
   нужного уровня → один Adapt → второй вызов уже cache hit).

## DoD
- Поведение URL-флоу, history-флоу и card-callback'а одинаковое
  (включая локализованное сообщение об ошибке regen).
- Дублирующийся код «load article + check ownership + count words +
  preview + ensure adapted level + render» живёт в одном месте.
- `make build` + `make test` зелёные.
