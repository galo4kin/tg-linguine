# Рефакторинг 30.5. Перевести articles-тесты на `mock.Provider`

## Контекст
Шаг 28 ввёл `internal/llm/mock.Provider` как канонический
fixture-driven стенд для `llm.Provider` (валидирует ответы через
реальную JSON-схему). Но в `internal/articles/` тесты до сих пор
используют свой приватный `stubLLM`:

- `internal/articles/usecase_test.go:29-51` — определение `stubLLM`,
  поля `calls`, `lastRequest`, `lastAdaptRequest`.
- Использования: `usecase_test.go:128, 180, 203, 229, 265, 370`,
  `long_test.go:87, 157, 196`.

Параллельные решения одной задачи — leaky abstraction. Хочется одну
точку правды.

## Цель
В тестах `internal/articles/` использовать `mock.Provider` (или
максимально близкий к нему стенд), удалить локальный `stubLLM`.

## Действия
1. Импортировать `internal/llm/mock` из `articles_test`.
2. На каждом сайте, где сейчас создаётся `&stubLLM{...}`,
   заменить на `mock.Provider` с эквивалентной настройкой:
   - `stubLLM{resp: sampleResponse()}` → `mock.Provider{AnalyzeResp: sampleResponse()}`
     (или через `mock.New()` + override, если фикстура подходит);
   - `stubLLM{err: llm.ErrRateLimited}` → `mock.Provider{AnalyzeErr: llm.ErrRateLimited}`;
   - тесты на Adapt — аналогично через `AdaptResp`/`AdaptErr`.
3. Где тесты читают `s.calls` / `s.lastRequest`, перейти на
   `len(p.AnalyzeCalls)` / `p.AnalyzeCalls[len-1]`. Аналогично для
   `adaptCalls` / `lastAdaptRequest`.
4. Удалить определение `stubLLM` из `usecase_test.go`.
5. Решить, где жить хелперу `sampleResponse()`. Если его проще
   заменить на `mock.LoadAnalyze("analyze_clean")` — заменить и
   убрать. Если тесту нужны точечные правки на полях — оставить как
   есть, но не делать `stubLLM` обратно.

## DoD
- В `internal/articles/` нет ни одного использования `stubLLM`.
- Тип `stubLLM` удалён из `usecase_test.go`.
- `make build` и `make test` зелёные.
- Поведение тестов не изменилось (количество кейсов то же,
  утверждения те же по смыслу).

## Зависимости
Шаги 28 (создание mock'а), 29 (последний кодовый шаг до окна).
