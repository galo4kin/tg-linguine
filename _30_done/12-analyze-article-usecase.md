# Шаг 12. Use case AnalyzeArticle + handler (5–7h)

## Цель
Сквозной сценарий: пользователь шлёт URL → бот извлекает текст →
зовёт LLM → парсит → сохраняет → отвечает article card.

## Контекст
Это первая «полезная» точка end-to-end. После этого шага можно
показывать другу.

## Действия
1. `internal/articles/usecase.go`: `AnalyzeArticle(ctx, userID, url
   string) (*Article, error)`.
   Шаги:
   - получить активный язык/уровень пользователя; если нет → ошибка
     `ErrNoActiveLanguage`;
   - получить расшифрованный API-ключ; если нет → `ErrNoAPIKey`;
   - извлечь статью (`Extractor`);
   - вызвать LLM (`Provider.Analyze`);
   - в одной транзакции — сохранить article + words + связи;
   - вернуть `*Article` с заполненной картой.
2. Handler сообщения с URL: `internal/telegram/handlers/url.go`.
   Регекс на URL в тексте сообщения. Если найден — запускает usecase.
3. UX-статусы (через `EditMessageText`):
   - сразу: «🔎 Загружаю статью…»,
   - после извлечения: «🧠 Анализирую…»,
   - финал: article card.
4. Article card (i18n строки): заголовок, обнаруженный CEFR, summary
   на изучаемом языке, превью первых 5 слов, кнопка «Показать все
   слова» (для шага 13).
5. Обработка ошибок: невалидный URL, paywall (ErrNotArticle),
   слишком длинная статья (отдельный шаг 26 — здесь возвращаем
   общую ошибку), LLM-ошибка → понятное сообщение пользователю.

## DoD
- Отправка валидного URL → через ~15–60s бот шлёт article card.
- Запись в `articles`, `article_words`, `dictionary_words` есть.
- Промежуточные статусы видны пользователю.
- При ошибке (например, ключ не задан) — понятное сообщение, без
  паники.
- Лог содержит `duration_ms`, `article_chars`, `words_count`.
- `make build` + `make test` зелёные.
